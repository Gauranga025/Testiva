Usage

How It Works — In 6 Steps

1

Connect Your GitHub Repository

Navigate to the workspace, click Connect Repository, and authorize Testiva via GitHub OAuth. Select the repository you want to test. Testiva stores your token securely and uses it only to fetch package.json and file structure.

2

Configure Target Domain

In Repository Settings, set the Target Domain: http://localhost:3000 for local dev, or your deployed URL. Optionally add Global Project Instructions — these are injected into every AI prompt (e.g., test credentials, dark-mode toggle location).

3

Generate Test Cases

Click Generate Test Cases. Testiva runs RepositoryIntelligenceService.analyzeRepository() and feeds structured context to Gemini 2.5 Flash. You get typed test cases with title, description, type, priority, target route, target files, and expected result.

4

Run a Test — Watch the Pipeline

Click Run on any test case. The execution modal opens and shows all 13 pipeline stages in real time — from health checks to DOM discovery to Playwright execution. Logs stream live from every stage.

5

Review Failure Analysis (if needed)

If a test fails, the AI Failure Analysis card appears with root cause, confidence score, broken selector, suggested replacement, step-by-step fix recommendation, and whether self-healing was triggered and succeeded.

6

Watch the Browserbase Recording

Every execution produces a full browser recording. Click Watch Recording to see exactly what Playwright did — clicks, navigation, form fills, and the moment of failure — at browserbase.com/sessions/{sessionId}.

Features

Key Capabilities

🧠

Repository Intelligence

Static analysis of package.json, file tree, env vars, and code files. Detects framework, auth provider, ORM, routing type, middleware, and architecture pattern — cached in PostgreSQL as jsonb.

🌐

Application Intelligence

Live Browserbase session scans every nav item, button, form, route, dialog, tab, dropdown, accessibility node, and login form. Returns a DomSummary guaranteed to reflect actual UI.

⚡

AI Playwright Generation

Gemini 2.5 Flash generates Playwright scripts grounded in real DOM and repository context. Scripts are syntax-checked via new Function() before execution. Llama 3.3 70B is the automatic fallback.

🔧

Self-Healing Tests

On UI-change failures, the AI generates a repaired script, re-executes it, and records the working selector in Repository Memory. Failed selectors are flagged to avoid future use.

🔍

AI Failure Analysis

Complete FailureContext collection — screenshot, console logs, network logs, Playwright context, execution timeline. Produces a FailureReport with root cause, confidence score, and fix recommendation.

🌍

Localhost via Cloudflare

Automatically spawns cloudflared, parses the public trycloudflare.com URL, verifies reachability, rewrites scripts, and tears down the tunnel post-execution. Zero manual setup.

♿

Accessibility Auditing

WCAG 2.1 AA compliance checks via live DOM: heading hierarchy, form label coverage, button accessibility, aria-label presence, and screen reader compatibility. Per-element violation reports.

📊

Coverage Analytics

Route coverage, API coverage, accessibility score, and visual regression score tracked by AnalyticsEngine across every execution run. Time-series history in Repository Memory.

🎯

Commit Impact Analysis

Maps changed files to affected components, routes, APIs, and test suites. Calculates a priority score and returns the minimal set of tests to run — saving CI time on every push.

Setup

Installation

Requirements

  • Node.js 18+
  • cloudflared CLI (for localhost testing)
  • Browserbase account
  • Neon PostgreSQL database
  • Clerk account
  • Google AI (Gemini) API key
  • Groq API key

Clone & Install

git clone https://github.com/Gauranga025/Testiva.git
cd Testiva/testiva
npm install
cp .env.example .env.local
# Fill in environment variables
npm run dev

Contributing

Branch Naming

Use the following convention:

feat/your-feature-name     # new features
fix/issue-description      # bug fixes
refactor/area-of-change    # refactoring
docs/what-you-documented   # documentation

Code Style Rules

  • TypeScript strict mode is enabled
  • All modules in lib/ export a class with typed public methods
  • New stages must be added to PipelineStageId and STATE_TRANSITIONS
  • All log lines must use formatLogLine(prefix, message) with a valid LogPrefix
  • New AI prompts added as static methods to PromptBuilders
